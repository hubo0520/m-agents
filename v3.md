第三阶段详细 PRD
1. 文档信息

产品名：商家经营保障 Agent V3
阶段定义：第三阶段 / 多 Agent + 生产化
产品形态：内部运营与审批工作台
目标用户：风险运营、融资运营、理赔运营、合规复核、管理员
技术主选：LangGraph + FastAPI + PostgreSQL + OpenAI Responses API
产品定位：把“单案件分析工具”升级为“可持续运行的多 Agent 风控执行系统”

2. 阶段目标

第三阶段的目标不是再多做几个页面，而是把系统升级成这 4 个能力：

多 Agent 协同
把监控、诊断、推荐、证据、执行、合规拆成明确职责的 Agent。

长流程可恢复
一条案件流转可以跨分钟、小时甚至天级暂停和恢复，支持人审、外部系统回调、失败重试。LangGraph 的 durable execution 和 checkpointer 正适合这一层。

强审批与可审计
高风险领域应保留人类复核，OpenAI 也明确建议在高风险场景中让人审查输出，并让审查者能看到原始依据。

生产级观测与评测
每一次 agent run、每个工具调用、每个审批、每次失败回退都可追踪、可复盘、可评测。

4. 核心产品原则
原则 1：代码主导流程，LLM 只在边界内工作

案件状态流转由代码控制

工具选择受 allowlist 约束

敏感动作前必须审批

LLM 输出必须为 schema 化结果

这和 OpenAI 官方现在强调的分工是一致的：应用代码负责 orchestration、tool loop、retries，prompt 负责行为边界与输出约束。

原则 2：所有输出必须可追溯

每条建议挂 evidence_ids

每条审批有 comment 和 reviewer

每次 agent run 有 trace_id / run_id / prompt_version / model_version

原则 3：金融动作默认保守

“建议可执行” 不等于 “自动执行”

涉及融资、反欺诈、拒赔、额度判断的动作默认进审批

原则 4：结构化优先

OpenAI 的 Structured Outputs 能保证输出遵守 schema，官方也建议能用时优先用它而不是 JSON mode。第三阶段所有 agent 间通信和落库结果都按 schema 走。

6. 业务场景范围

第三阶段覆盖 4 条主场景：

场景 A：退货激增 -> 现金缺口 -> 回款加速

发现商家退货异常，上升到案件，预测 14 日缺口，推荐回款加速，人工审批后创建执行任务。

场景 B：退货异常 -> 疑似欺诈 -> 人工复核

发现异常退货模式，自动拉证据，生成复核任务，等待风控审核。

场景 C：缺口持续 -> 经营贷草稿

经营健康度达标时，生成贷款申请草稿与材料包，不自动提交。

场景 D：保险 / 理赔材料编排

根据保单规则与订单、物流、退款记录生成理赔草稿，进入人工确认。

8. Agent 架构设计
8.1 总体选型

主编排：LangGraph
原因：显式图式编排、durable execution、暂停/恢复、人审、状态持久化、调试能力，和你的金融场景天然匹配。

模型调用：OpenAI Responses API
原因：支持 stateful interactions、built-in tools，reasoning models 更适合 multi-step agentic workflows，而且 Structured Outputs 能把 agent 输出钉死在 schema 上。

A1 Monitor Agent

10. 功能需求
FR-301 多 Agent 编排引擎

系统必须支持多个 specialist agents 通过显式 graph 协同执行。

验收：

至少 6 个 agent 节点可运行

支持条件分支

支持节点失败后重试或人工接管

FR-302 工作流持久化与恢复

每个 workflow run 必须可落库、可恢复、可回放。

验收：

服务重启后 run 可恢复

审批完成后能从原中断点继续

节点结果不可丢失

FR-303 审批门禁

所有敏感动作必须进入审批队列。

敏感动作包括：

经营贷建议进入执行阶段

反欺诈复核结论

理赔草稿提交前动作

回款加速申请

验收：

未审批不得执行

驳回后 workflow 进入 REJECTED 或 REWORK

审批意见写入审计日志

FR-304 工具与连接器注册中心

系统必须支持统一工具注册与权限策略。

工具类别：

数据查询工具

规则判断工具

证据检索工具

外部执行工具

审计工具

验收：

每个工具具备 tool_name、version、approval_policy

工具调用日志可查询

工具支持幂等键

FR-305 证据绑定

每条 root cause 与 recommendation 必须绑定 evidence_ids。

验收：

建议无证据时不得进入执行阶段

证据点击可展开源数据

导出时带证据清单

FR-306 审批中心

审批中心支持待办查看、审批、驳回、修改后批准。

验收：

审批后自动恢复 workflow

审批记录含 reviewer、time、comment、final_action_json

支持 SLA 超时提醒

FR-307 工作流运行中心

展示运行中的 graph、节点耗时、错误、回放入口。

验收：

可查看每个 run 的当前节点

可查看失败原因

可对 retryable 任务执行 retry

FR-308 模型 / Prompt / Schema 版本管理

所有 agent 输出都必须绑定模型版本、prompt 版本、schema 版本。

验收：

任意案件可追溯到版本信息

新版本支持灰度开关

可以回滚到旧版本

FR-309 评测中心

支持离线评测与线上抽样评测。

验收：

可创建 eval dataset

可跑指定版本的建议质量评测

输出采纳率、回退率、证据覆盖率、schema 合格率

FR-310 失败回退

任何 agent 节点失败时，系统进入可控回退，而不是整条链路崩掉。

验收：

LLM 节点失败可回退到规则建议模式

外部工具失败可进入人工处理

所有失败写入 audit_logs 与 workflow_runs

FR-311 权限控制

不同角色只能访问允许的案件、审批和配置项。

验收：

合规角色可看全部审计，但不可直接改业务建议

普通运营不可改模型策略

管理员可配置审批规则

FR-312 案件重开与再分析

已完成或驳回案件支持重开。

验收：

重开产生新版本 workflow_run

旧版本保留，不可覆盖

新版本与旧版本差异可比对

14.1 必审动作

14.2 守卫规则

无 evidence_ids 的建议不得执行

schema 校验失败直接阻断

命中禁止语义的输出直接阻断

超出额度阈值的动作必须升级审批

模型输出不得直接写业务最终结论字段

14.3 工具级守卫

OpenAI Agents SDK 官方就把 input / output / tool guardrails 做成一等概念，说明“敏感工具调用前后都要拦截”是行业内已经成型的做法；即便你主编排选 LangGraph，我也建议把这个思想照搬进你自己的工具网关里。

17. 第三阶段技术落地约束

给 AI coding 时要明确这几个约束：

LangGraph 只做编排，不做业务计算

所有节点输入输出都用 Pydantic

所有 LLM 输出必须走 Structured Outputs

所有敏感工具都有 approval_policy

所有 run / node / tool / approval 都有唯一 ID

数值指标来自 domain service，不来自 LLM

workflow state 必须可序列化

任何节点都不得直接覆盖历史版本
