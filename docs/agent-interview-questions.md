## 一、Agent 核心概念与设计模式（6 题）

### Q1：请对比 ReAct、Plan-and-Execute、Multi-Agent Pipeline 三种 Agent 架构模式的适用场景 ⭐⭐

**考察要点**：候选人是否真正理解不同 Agent 模式的本质差异，而非只会用某个框架。

**回答思路**：
- **ReAct**（Thought-Action-Observation 循环）：适合开放性探索任务，Agent 自主决定下一步调用什么工具。优势是灵活性高，劣势是执行路径不可预测、不便于审计和降级。
- **Plan-and-Execute**（先规划再执行，支持动态 re-plan）：适合复杂但半结构化任务，Planner 生成计划后 Executor 逐步执行。优势是有全局视野，劣势是 re-plan 增加不确定性。
- **Multi-Agent Pipeline**（预定义工作流 + 条件路由）：适合流程明确的领域任务，每步可追踪、可审计。优势是确定性强，劣势是灵活性有限。
- **结合项目**：我选择了 Multi-Agent Pipeline + Conditional Routing，因为风控分析是流程明确的领域任务（分诊→取证→诊断→建议→审批→执行），需要确定性和可审计性。同时在 Triage Agent 的 `recommended_path` 中引入了"轻量级规划"的意味——分诊结果决定后续走哪条路径。

---

### Q2：请深入解释 ReAct 模式的 Thought-Action-Observation 循环机制，以及它的局限性 ⭐⭐

**考察要点**：候选人是否理解 ReAct 的推理循环细节，以及在生产环境中的实际限制。

**回答思路**：
- **循环机制**：Agent 先 Thought（思考当前状态，决定下一步策略）→ Action（选择并调用一个工具/API）→ Observation（获取工具返回结果）→ 再次 Thought... 直到得出最终答案。
- **局限性**：① 循环次数不可控，可能陷入无限循环或过早终止；② 每次循环都消耗 Token，成本不可预测；③ 执行路径随机性高，同一输入可能产生不同路径，不利于审计；④ 降级策略极难设计——你不知道 Agent 下一步要做什么，无法预置降级逻辑。
- **结合项目**：这些局限性正是我不选择 ReAct 作为整体架构的原因。但我在 Diagnosis Agent 内部使用了 CoT 推理链（Chain-of-Thought），引导 LLM 按"数据特征分析→风险因子关联→根因结论"的步骤推理，这借鉴了 ReAct 的 Thought 环节但避免了 Action 的不确定性。

---

### Q3：Reflection/Self-Critique 模式在 Agent 系统中如何应用？你的项目中有类似设计吗？ ⭐⭐⭐

**考察要点**：候选人是否理解 Reflection 模式的核心思想（自我评估→改进循环），以及是否能在实际项目中识别和应用变体。

**回答思路**：
- **标准 Reflection**：同一个 Agent 生成输出后，用另一个 Prompt 评估输出质量，再根据反馈重新生成（Generator→Critic→Re-generate 循环）。
- **项目中的变体**：
  - **Cross-Agent Validation**：Compliance Guard 对上游 Recommend Agent 的输出做合规校验——不是自我反思，而是引入了"第二视角"，类似 Code Review。如果校验不通过，会阻断工作流。
  - **LLM-as-Judge 评测中心**：用独立的 LLM 评估分析质量（综合评分、幻觉检测、证据覆盖率），本质上是一种离线 Reflection 机制。
- **区别**：标准 Reflection 是同步的、同 Agent 内的；我的设计是跨 Agent 的（Compliance Guard）和异步的（评测中心），更适合生产环境的质量保障。

### Q4：Tool Use 与 Function Calling 在 Agent 系统中如何设计？什么时候应该让 LLM 自主选择工具，什么时候应该预定义？ ⭐⭐

**考察要点**：候选人是否理解工具调用的两种范式——LLM 自主选择 vs 工作流预定义，以及各自的适用场景。

**回答思路**：
- **LLM 自主选择工具**（ReAct 范式）：将工具描述（JSON Schema）注入 System Prompt，LLM 通过 Function Calling 决定调什么工具、传什么参数。适合工具集较大、任务不确定的场景。
- **工作流预定义工具**（Pipeline 范式）：每个节点调用哪些工具是代码中写死的，LLM 只负责推理，不负责选工具。适合流程明确、需要精确控制的场景。
- **结合项目**：我的 7 个 Agent 都是工作流预定义调用——Evidence Agent 查哪些数据源、Analysis Agent 调哪个 LLM、Execution Agent 创建什么任务，都是代码级确定的。原因是：① 风控场景需要精确控制工具调用的权限和超时；② 自建工具注册中心可以统一管理降级策略。如果未来引入 ReAct，会考虑将部分工具迁移为 LangChain Tools 或 MCP Tools 格式。

### Q5：Agent 的 Memory 机制有哪些类型？在多 Agent 系统中如何管理上下文？ ⭐⭐

**考察要点**：候选人是否理解 Agent Memory 的分类（短期/长期/外部），以及多 Agent 间上下文传递的工程挑战。

**回答思路**：
- **Memory 分类**：
  - **短期记忆**（Working Memory）：当前对话/任务的上下文窗口，受 Token 限制。
  - **长期记忆**（Long-term Memory）：持久化存储的历史信息，如向量数据库中的文档、数据库中的历史案件。
  - **外部记忆**（External Memory）：通过 RAG 检索的外部知识。
- **结合项目**：
  - **短期记忆**：`analysis_context` 机制实现 Agent 间洞察接力传递，单条上限 200 字符，总上下文上限 1500 字符，超限 FIFO 淘汰——这是在 Token 成本和信息传递之间的工程权衡。
  - **长期记忆**：案件分析完成后自动向量化存入 ChromaDB，支持后续 RAG 对话追问。
  - **状态记忆**：`GraphState` 作为 LangGraph 工作流的全局状态，14 个节点间传递完整的分析中间结果。

### Q6：什么是 Agent 的 Hallucination（幻觉）问题？在你的项目中如何检测和缓解？ ⭐

**考察要点**：候选人是否理解 LLM 幻觉在 Agent 系统中的危害，以及工程化的检测和缓解手段。

**回答思路**：
- **幻觉定义**：LLM 生成看似合理但实际错误或无依据的内容。在 Agent 系统中尤其危险——幻觉会通过 Agent 链传播放大。
- **结合项目**：
  - **Structured Output 约束**：通过 Pydantic v2 Schema 强制 Agent 输出结构化数据，约束输出格式减少自由发挥空间。
  - **Cross-Agent Validation**：Compliance Guard 校验 Recommend Agent 的建议是否违反禁止性规则，拦截明显不合理的输出。
  - **LLM-as-Judge 幻觉检测**：评测中心专门设有幻觉检测维度，评估分析结论是否有证据支撑。
  - **证据覆盖率**：评测维度之一，检查分析结论引用的证据比例，无证据支撑的结论会被标记。
  - **人在回路**：审批闭环是最终防线——AI 的处置建议必须人工审批后才能执行。

## 二、LLM 工程化（5 题）

### Q7：如何保障 LLM 输出的结构化可靠性？Structured Output 失败时怎么办？ ⭐⭐

**考察要点**：候选人是否有 LLM 输出不可靠的实战经验，以及是否设计了降级方案。

**回答思路**：
- **核心挑战**：LLM 的输出本质上是概率性的，即使指定了 JSON Schema，也可能输出不合法的 JSON、缺少必填字段或类型不匹配。
- **结合项目——双降级策略**：
  1. **L1 Structured Output**：优先使用 `beta.parse()` API，利用模型原生的结构化输出能力，直接返回 Pydantic 对象。
  2. **L2 JSON Mode 降级**：`beta.parse()` 失败后，自动切换到 `response_format=json_object` 模式，解析 JSON 后再通过 Pydantic v2 Schema 校验。
  3. **L3 异常兜底**：JSON 解析也失败时，进入节点级的三级降级流程（自动重试→规则引擎→人工接管）。
- **关键设计**：每个 Agent 的输入/输出都定义了 Pydantic Schema 作为数据契约，这不仅保障了结构化输出，还实现了 Agent 间的类型安全传递。

---

### Q8：Prompt Engineering 有哪些最佳实践？你是如何管理和迭代 Prompt 的？ ⭐

**考察要点**：候选人对 Prompt 工程的理解深度——不只是会写 Prompt，还要会管理 Prompt 的生命周期。

**回答思路**：
- **最佳实践**：① 角色定义清晰（System Prompt 明确 Agent 身份和职责）；② 输出格式约束（JSON Schema + 示例）；③ Few-shot 示例提升一致性；④ 链式思维引导（CoT）提升推理深度。
- **结合项目——Prompt 灰度分流机制**：
  - **痛点**：Prompt 修改可能导致输出质量波动，全量发布风险太大。
  - **方案**：支持新版 Prompt 按流量比例灰度上线（如先给 10% 流量试水），通过 LLM-as-Judge 评测中心对比新老版本的分析质量（综合评分、幻觉检测、证据覆盖率），验证通过再全量发布。
  - **效果**：实现了 Prompt 的可灰度、可回滚、可评测的完整生命周期管理。

---

### Q9：多 Agent 系统中 Token 消耗如何管理？如何控制成本？ ⭐⭐

**考察要点**：候选人是否关注 LLM 调用的成本问题，以及是否有实际的成本控制策略。

**回答思路**：
- **Token 消耗来源**：① System Prompt（每次调用都会消耗）；② 上下文传递（Agent 链越长，累积上下文越大）；③ 输出长度（结构化输出通常比自由文本更可控）。
- **结合项目——累积式上下文的 FIFO 淘汰**：
  - **问题**：7 个 Agent 链式传递，如果每个 Agent 的完整输出都注入下游 Prompt，Token 消耗会指数增长。
  - **方案**：`analysis_context` 机制——每个 Agent 只提取关键洞察（单条上限 200 字符），累积传递给下游 Agent（总上下文上限 1500 字符），超限时按 FIFO 淘汰最早的上下文。
  - **另一个策略**：不是所有 Agent 都需要调用 LLM——Evidence Agent（证据收集）、Compliance Agent（合规校验）、Execution Agent（执行落地）是纯规则驱动的，不消耗 LLM Token。7 个 Agent 中只有 4 个依赖 LLM。

---

### Q10：LLM 服务不可用时，Agent 系统如何保证可用性？ ⭐⭐⭐

**考察要点**：候选人是否有生产级 Agent 系统的容灾设计经验，这是区分"Demo 级"和"生产级"的关键。

**回答思路**：
- **核心问题**：LLM 是 Agent 系统的"大脑"，一旦不可用，整个系统可能瘫痪。
- **结合项目——三级降级容灾**：
  1. **L1 自动重试**：指数退避（1s→2s→4s，最多 3 次），覆盖全部 14 个工作流节点。应对临时网络抖动和 API 限流。
  2. **L2 规则引擎降级**：仅限 LLM 节点——LLM 不可用时，Triage Agent 用预置规则判断案件类型，Diagnosis Agent 用规则引擎做基本风险评估，Recommend Agent 按规则生成标准建议。风控引擎层（Layer 5）完全不依赖 LLM。
  3. **L3 人工接管**：创建人工接管工单，工作流暂停等待人工处理。这是最终兜底。
- **关键设计**：非 LLM Agent（Evidence、Compliance、Execution）天然不受 LLM 故障影响，系统架构本身就分离了"需要智能"和"不需要智能"的环节。

---

### Q11：如何评估 Agent 系统的输出质量？你是怎么做评测的？ ⭐⭐

**考察要点**：候选人是否有 Agent 系统质量保障的意识和实践。

**回答思路**：
- **评测挑战**：Agent 的输出是自然语言，不像传统软件可以用 assert 判断对错。
- **结合项目——LLM-as-Judge 评测中心**：
  - **方案**：用独立的 LLM 当"裁判"，自动评估分析质量。
  - **评测维度**：综合评分（分析是否全面准确）、幻觉检测（结论是否有证据支撑）、证据覆盖率（引用了多少比例的可用证据）。
  - **用途**：① Prompt 灰度对比——新老 Prompt 的质量差异量化；② 持续监控——生产环境中定期抽样评测；③ 回归测试——代码变更后验证输出质量不退化。
- **补充**：除了 LLM-as-Judge，还可以结合人工评测（抽样标注 + Cohen's Kappa 一致性评估），但成本较高。

## 三、RAG 系统设计（4 题）

### Q12：请描述一个完整的 RAG 系统架构，从数据准备到检索再到生成 ⭐

**考察要点**：候选人是否理解 RAG 的端到端流程，而非只知道"向量检索"。

**回答思路**：
- **完整流程**：① 数据准备（文档收集→清洗→切片）→ ② 向量化（Embedding 模型将文本转为向量）→ ③ 索引（存入向量数据库，建立检索索引）→ ④ 检索（用户查询向量化→相似度检索→Top-K 召回）→ ⑤ 注入（召回内容注入 LLM Prompt）→ ⑥ 生成（LLM 基于增强上下文生成回答）。
- **结合项目**：
  - 数据准备：案件分析完成后，自动将分析结果按 6 种文档类型（案件摘要、根因分析、动作建议、现金流预测、证据项、商家信息）切片。
  - 向量化：使用 text-embedding-v4（1024 维）。
  - 索引：存入 ChromaDB，使用 HNSW 索引。
  - 检索：余弦相似度 Top-5 召回。
  - 生成：召回内容注入 Prompt 后，LLM 通过 SSE 流式推送回答。

---

### Q13：文档切片有哪些策略？固定长度切片和语义切片各有什么优劣？ ⭐⭐

**考察要点**：候选人是否理解切片策略对 RAG 检索质量的影响，以及如何根据数据特点选择策略。

**回答思路**：
- **固定长度切片**：按字符/Token 数切分（如每 512 Token 一片）。优点：实现简单、切片大小均匀。缺点：可能切断语义单元，导致召回的上下文不完整。
- **语义切片**：按内容的语义边界切分（如按段落、按主题）。优点：保证每片的语义完整性。缺点：切片大小不均匀，实现复杂度高。
- **结合项目——6 种文档切片策略**：
  - 按语义单元切片而非固定长度：案件摘要（完整一片）、根因分析（每条独立成片）、动作建议（每条独立成片）、现金流预测（完整一片）、证据项（每条独立成片）、商家信息（完整一片）。
  - 每案件典型切片数 10~20 片。
  - **为什么这样设计**：用户追问"退货率为什么飙升"时，召回的应该是完整的一条根因分析，而不是被切断的半句话。

---

### Q14：Embedding 模型选型有哪些考量？如何优化检索质量？ ⭐⭐

**考察要点**：候选人是否关注 Embedding 模型的选型细节，以及是否有检索质量优化的实际经验。

**回答思路**：
- **选型考量**：① 维度（影响检索精度和存储成本）；② 语义空间对齐（Embedding 模型与 LLM 是否同系列）；③ 多语言支持；④ 推理速度和成本；⑤ 上下文窗口长度。
- **结合项目**：
  - 选择 text-embedding-v4（1024 维），与推理 LLM（通义千问 qwen-plus）同属通义系列，语义空间对齐度高。
  - **检索质量优化**：① 语义切片保证召回完整上下文；② Top-5 召回兼顾精度和覆盖率；③ HNSW 索引提供近似最近邻的高效检索。
- **可扩展方向**：如果检索质量不够，可以引入 Reranker（二次排序模型），先粗召回 Top-20 再精排到 Top-5。

---

### Q15：RAG 系统如何做降级和容错？向量数据库不可用怎么办？ ⭐⭐⭐

**考察要点**：候选人是否在 RAG 系统中也考虑了生产级的容错设计。

**回答思路**：
- **核心问题**：RAG 链路上任何环节故障（Embedding 服务、向量数据库、LLM）都会导致对话功能不可用。
- **结合项目——RAG 降级链**：
  1. **正常路径**：text-embedding-v4 向量化 → ChromaDB HNSW 检索 → Top-5 召回 → 注入 Prompt → LLM 生成。
  2. **L1 降级**：text-embedding-v4 不可用 → 降级到 ChromaDB 默认嵌入模型。
  3. **L2 降级**：ChromaDB 完全不可用 → 将分析结果全量注入 Prompt（不走检索，直接把所有内容塞进去）。
  4. **效果**：对话功能在任何异常下都可用，只是回答的精准度可能下降。
- **设计哲学**：宁可回答质量稍降，也不能让功能完全不可用——这是生产级系统和 Demo 的核心区别。

## 四、多 Agent 编排与协作（5 题）

### Q16：LangGraph 的 StateGraph 和传统工作流引擎（如 Airflow）有什么区别？为什么选 LangGraph？ ⭐⭐

**考察要点**：候选人是否理解 LangGraph 的定位——它不只是一个工作流引擎，而是专为 LLM Agent 设计的编排框架。

**回答思路**：
- **核心区别**：
  - **Airflow**：面向数据管道/ETL，DAG 中每个节点是一个"任务"，任务间通过 XCom 传递数据，不关注 LLM 交互。
  - **LangGraph**：面向 LLM Agent 编排，`StateGraph` 中每个节点可以调用 LLM，节点间通过 `GraphState` 传递结构化状态，原生支持条件路由、断点恢复。
- **结合项目**：选择 LangGraph 的原因：
  - ① 原生 `GraphState` 管理，14 个节点间的状态传递代码量极小。
  - ② 条件边（Conditional Edge）天然支持分诊后的路径分支。
  - ③ 断点机制天然支持审批闭环——工作流在审批节点暂停，审批通过后恢复。
  - ④ 与 LangChain 生态兼容，未来引入 LangChain Tools 无缝对接。

---

### Q17：多 Agent 之间如何传递上下文？如何避免上下文爆炸？ ⭐⭐⭐

**考察要点**：候选人是否处理过多 Agent 链式传递时上下文膨胀的问题——这是一个非常实际的工程问题。

**回答思路**：
- **上下文爆炸问题**：7 个 Agent 链式执行，如果每个 Agent 的完整输出都传给下游，Prompt 会越来越长，Token 成本指数增长，甚至超出模型上下文窗口。
- **结合项目——累积式上下文 + FIFO 淘汰**：
  - `GraphState` 传递完整的结构化分析结果（Agent 间精确传递所需数据）。
  - `analysis_context` 传递关键洞察摘要（给 LLM 看的上下文）：每个 Agent 只提取 1-3 条关键洞察，每条上限 200 字符，总上下文上限 1500 字符，超限时 FIFO 淘汰最早的条目。
  - **双通道设计**：精确数据走 `GraphState`（代码逻辑消费），摘要洞察走 `analysis_context`（LLM Prompt 消费），各取所需。

### Q18：条件路由在 Agent 工作流中如何设计？你的项目有几个条件路由节点？ ⭐

**考察要点**：候选人是否理解条件路由的必要性，以及如何在 LangGraph 中实现。

**回答思路**：
- **条件路由的作用**：根据上游节点的输出动态决定下游走哪条路径，实现工作流的动态分支。
- **结合项目——3 个条件路由节点**：
  1. **分诊路由**：Triage Agent 判断案件类型（现金缺口/疑似欺诈/融资/理赔）后，根据 `recommended_path` 决定后续处理路径。
  2. **合规校验路由**：Compliance Guard 校验建议的合规性后，决定放行（继续生成报告）或阻断（标记不合规，需要人工复核）。
  3. **审批路由**：审批通过后继续执行落地，审批拒绝则终止流程。
- **实现方式**：LangGraph 的 `add_conditional_edges()` API，传入路由函数，函数读取 `GraphState` 中的判断结果返回目标节点名称。

### Q19：工作流的断点恢复是怎么实现的？审批等待期间工作流状态存在哪里？ ⭐⭐⭐

**考察要点**：候选人是否理解长时工作流的断点恢复机制——这不仅是 Agent 问题，也是分布式系统设计问题。

**回答思路**：
- **断点恢复的必要性**：审批可能需要数小时甚至数天，工作流不能一直占用内存等待。
- **结合项目**：
  - 工作流执行到审批节点时，将当前 `GraphState` 和工作流进度持久化到 MySQL 数据库，然后工作流暂停释放资源。
  - 审批人操作后（通过/拒绝），系统从数据库恢复 `GraphState`，从断点继续执行后续节点（如 Execution Agent 执行落地）。
  - **关键设计**：`GraphState` 必须是完全可序列化的（所有字段都是 Pydantic 模型或基础类型），否则无法持久化和恢复。

### Q20：Orchestrator 编排器模式和去中心化 Agent 协作模式各有什么优劣？ ⭐

**考察要点**：候选人是否理解不同的 Agent 协作模式——中心化 vs 去中心化。

**回答思路**：
- **Orchestrator 模式**（中心化）：一个编排器统一调度所有 Agent，控制执行顺序和数据流。优点：流程清晰、便于监控和调试。缺点：编排器是单点，逻辑复杂时难以维护。
- **去中心化模式**：Agent 之间直接通信，通过消息传递协作。优点：灵活、可扩展。缺点：协作行为难以预测、难以审计。
- **结合项目**：选择了 Orchestrator 模式。原因：
  - ① 风控分析需要严格的执行顺序（不能在诊断之前就给建议）。
  - ② 三级降级需要编排器统一管理每个节点的失败处理。
  - ③ 审批闭环需要编排器控制工作流的暂停和恢复。
  - ④ 日志审计需要编排器记录完整的执行轨迹。

## 五、生产级 Agent 系统（5 题）

### Q21：Agent 系统的容错设计应该从哪些维度考虑？ ⭐⭐

**考察要点**：候选人是否有全面的容错设计思维——不只是"LLM 挂了怎么办"，还包括其他维度。

**回答思路**：
- **容错维度**：
  1. **LLM 服务故障**：超时、限流、不可用。
  2. **输出质量异常**：幻觉、格式错误、逻辑矛盾。
  3. **外部依赖故障**：数据库、向量数据库、第三方 API。
  4. **数据异常**：输入数据缺失、格式错误、超出预期范围。
  5. **长时任务中断**：进程重启、网络中断、审批超时。
- **结合项目**：
  - LLM 故障 → 三级降级（自动重试→规则引擎→人工接管），覆盖全部 14 节点。
  - 输出质量 → Structured Output 双降级 + Cross-Agent Validation + LLM-as-Judge。
  - 外部依赖 → RAG 降级链（ChromaDB 不可用时降级到全量注入 Prompt）。
  - 长时中断 → 断点恢复（`GraphState` 持久化到 MySQL，审批后恢复执行）。

### Q22：Agent 系统的可观测性如何设计？你是如何做日志和监控的？ ⭐

**考察要点**：候选人是否关注 Agent 系统的可观测性——Agent 的行为是非确定性的，可观测性比传统系统更重要。

**回答思路**：
- **可观测性三支柱**：日志（Logging）、指标（Metrics）、追踪（Tracing）。
- **结合项目**：
  - **日志**：使用 Loguru 框架，双 Sink（控制台 + 文件），按总大小轮转清理。每个 Agent 节点的输入、输出、耗时、降级事件都记录日志。
  - **监控**：Docker Compose 集成 Loki + Grafana，Loki 聚合日志，Grafana 可视化展示 Agent 工作流的执行状态、降级触发频率、LLM 调用耗时等关键指标。
  - **追踪**：每个案件分析有唯一的 `case_id`，所有 Agent 的日志都关联此 ID，支持完整链路追踪。
- **Agent 特有的监控点**：LLM 调用成功率、Structured Output 降级率、平均 Token 消耗、RAG 检索命中率、端到端分析耗时。

### Q23：Agent 系统中的"人在回路"（Human-in-the-Loop）如何设计？为什么重要？ ⭐⭐

**考察要点**：候选人是否理解 AI 系统中人类监督的必要性，以及如何在工程层面实现。

**回答思路**：
- **为什么重要**：LLM 的输出是概率性的，在高风险决策场景（如金融风控）中，不能完全信任 AI 的判断。
- **结合项目——审批闭环 + RBAC**：
  - **审批闭环**：AI 的处置建议（融资申请、理赔、人工复核）不会直接执行，必须经过对应角色的审批后才能落地。工作流在审批节点自动暂停。
  - **RBAC 权限控制**：5 种角色 × 18 种权限矩阵，不同角色只能看到和操作自己权限范围内的功能。例如"执行审批"权限只有特定角色才拥有。
  - **Compliance Guard**：合规审查 Agent 是自动化的"人在回路"——在建议到达人工审批之前，先过一道自动化的合规检查。
- **设计层次**：自动化检查（Compliance Guard）→ 人工审批（审批闭环）→ 权限控制（RBAC），三层保障。

### Q24：如果你要对 Agent 系统做灰度发布，会怎么设计？ ⭐⭐⭐

**考察要点**：候选人是否理解 AI 系统灰度发布的特殊性——不只是代码灰度，还有 Prompt/模型灰度。

**回答思路**：
- **AI 系统灰度的特殊性**：传统软件灰度只关注代码变更，AI 系统还需要考虑 Prompt 变更和模型变更（换模型/换版本）。
- **结合项目——Prompt 灰度分流**：
  - 支持新版 Prompt 按流量比例灰度上线（如 10% 流量走新 Prompt，90% 走老 Prompt）。
  - 通过 LLM-as-Judge 评测中心对比新老版本的分析质量（综合评分、幻觉检测、证据覆盖率）。
  - 验证通过再逐步加大比例，最终全量发布。
  - 如果质量下降，支持一键回滚到老版本。
- **可扩展方向**：如果未来需要换 LLM 模型（如从 qwen-plus 换到 qwen-max），同样可以用灰度分流机制——10% 流量走新模型，对比质量后再全量切换。

### Q25：MCP（Model Context Protocol）协议是什么？如何在 Agent 系统中应用？ ⭐⭐

**考察要点**：候选人是否了解 Agent 生态中的新兴标准化协议。

**回答思路**：
- **MCP 核心理念**：Anthropic 提出的标准化协议，定义了 Agent 与外部工具/数据源之间的通信标准。核心是 MCP Server（提供工具）和 MCP Client（调用工具），工具通过标准 JSON Schema 描述。
- **核心价值**：① 工具描述标准化，跨框架复用；② 动态工具发现，Agent 可以在运行时发现新工具；③ 统一的调用接口，简化集成。
- **结合项目——如何引入 MCP**：
  - 将现有自建工具注册中心改造为 MCP Server，每个工具提供标准 JSON Schema 描述。
  - Orchestrator 作为 MCP Client，通过标准协议发现和调用工具。
  - 好处：工具可以跨 Agent 框架复用，未来如果引入第三方 Agent 框架也能无缝对接。
  - **权衡**：目前项目工具数量有限且变动不频繁，自建注册中心已够用。引入 MCP 的收益需要在工具规模扩大后才能体现。

## 六、系统设计开放题（3 题）

### Q26：如果让你从零设计一个多 Agent 客服系统，你会怎么设计架构？ ⭐⭐⭐

**考察要点**：候选人的全局架构设计能力——是否能快速抽象问题、拆分 Agent、设计协作模式。

**回答思路**：
- **需求分析**：先明确场景——售前咨询、售后投诉、技术支持等不同类型的客服需求。
- **Agent 拆分**（参考本项目的拆分思路）：
  - **路由 Agent**（类似 Triage）：识别用户意图，将请求路由到对应专业 Agent。
  - **知识问答 Agent**（RAG 驱动）：基于产品知识库回答常见问题。
  - **工单 Agent**（类似 Execution）：无法解决的问题创建工单转人工。
  - **情绪分析 Agent**：检测用户情绪，紧急/愤怒时优先转人工。
  - **摘要 Agent**（类似 Summary）：生成对话摘要，帮助人工客服快速了解上下文。
- **架构选择**：Multi-Agent Pipeline + 条件路由（和本项目类似），而非 ReAct——客服系统需要可控的对话流程，不能让 Agent 随意发挥。
- **关键设计**：降级到人工（类似三级降级）、对话历史管理（类似 Memory）、回答质量评测（类似 LLM-as-Judge）。

### Q27：如何将一个单 Agent 系统演进为多 Agent 系统？演进过程中有哪些关键决策点？ ⭐⭐⭐

**考察要点**：候选人的架构演进思维——不是一步到位，而是逐步演进。

**回答思路**：
- **演进路径**（参考本项目 V1→V5 的迭代经验）：
  1. **V1 单 Agent**：一个大 Prompt 做所有事情，快速验证可行性。
  2. **V2 拆分职责**：识别出可以独立的子任务，拆分为多个专业 Agent。
  3. **V3 引入编排器**：Agent 数量增多后，需要统一的编排逻辑管理执行顺序和数据流。
  4. **V4 加入容错**：生产化需求——降级、重试、断点恢复、人工接管。
  5. **V5 质量保障**：评测中心、灰度分流、监控告警。
- **关键决策点**：
  - **什么时候拆**：当一个 Prompt 超过 2000 Token 或职责超过 3 个时考虑拆分。
  - **怎么拆**：按业务领域拆（不是按技术功能拆），每个 Agent 有明确的输入/输出契约。
  - **要不要引入 LLM**：不是所有 Agent 都需要 LLM——纯规则逻辑的 Agent（如 Evidence、Compliance、Execution）不需要。

### Q28：Agent 系统的性能瓶颈通常在哪里？如何优化？ ⭐

**考察要点**：候选人是否关注 Agent 系统的性能问题，以及是否有实际的优化经验。

**回答思路**：
- **性能瓶颈分析**：
  1. **LLM 调用延迟**（最大瓶颈）：每次 LLM 调用 1-10 秒，7 个 Agent 中 4 个调用 LLM，串行执行时总延迟叠加。
  2. **向量化延迟**：案件分析完成后的切片 + Embedding 过程。
  3. **数据库查询**：Evidence Agent 需要跨多表查询。
- **优化策略**（结合项目）：
  - **并行化**：无依赖关系的 Agent 可以并行执行（如 Metrics 计算和 Forecast 预测可以并行）。
  - **SSE 流式输出**：用户不用等完整分析完成，RAG 对话通过 SSE 实时推送。
  - **非 LLM Agent**：将不需要推理的环节设计为纯规则 Agent（Evidence、Compliance、Execution），避免不必要的 LLM 调用。
  - **上下文精简**：FIFO 淘汰机制控制传给 LLM 的上下文长度，减少 Token 消耗和推理耗时。
